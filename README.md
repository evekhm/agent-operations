# Agent Operations

## Overview

The **Observability Agent** is an autonomous, AI-powered Site Reliability Engineer (SRE) designed specifically to analyze data generated by the **BigQuery Agent Analytics plugin**.

### The Vision & End Goal

In the era of the "Agentic Shift," traditional monitoring focused solely on reactive system metrics (like CPU or simple telemetry) is insufficient for non-deterministic AI applications. Agentic systems require proactive observation of emergent behaviors, tool execution pathways, and reasoning loops.

The end goal for the Observability Agent is to act as a **fully autonomous, prescriptive SRE** capable of comprehensive performance, cost, and safety monitoring. It aims to move beyond static dashboards by deeply integrating with the rich dataset of **ADK events and execution contexts** captured by the BigQuery plugin (tracking entire LLM reasoning traces, tool executions, state deltas, and multi-agent orchestration overhead). Ultimately, the agent will not only diagnose performance issues but proactively auto-remediate them, secure the system against prompt injections, score interaction quality, and feed high-quality "Golden Datasets" back into evaluation pipelines for continuous model fine-tuning.

See [high_level_observability_agent_architecture](project_context/design_docs/high_level_observability_agent_architecture.md) for more details.

### Current Capabilities

Currently, the Observability Agent operates at a diagnostic and early predictive level, acting as the analytical brain on top of the BigQuery Agent Analytics plugin's data. It executes specialized, data-driven **Playbooks** (`health`, `incident`, `latest`, `trend`, and `overview`) to investigate the true performance of your agentic applications across three architectural layers: the Agent Layer, the LLM Layer, and the Tool Layer.

Equipped with a comprehensive suite of BigQuery analytics tools, it can presently:
- Isolate LLM reasoning latency from external tool execution bottlenecks.
- Mathematically prove concurrency and detect sequential bottlenecks.
- Establish dynamic performance baselines using the fastest historical queries.
- Perform basic root cause synthesis on failed or slow requests.
- Produce highly structured, data-backed markdown reports outlining what is wrong and providing prioritized recommendations.

### Future Roadmap (To Be Built)

Based on the strategic design document, the following advanced capabilities and extensibility goals are still under development to realize the full vision:
- **Advanced Hypothesis Testing**: Performing rigorous root cause synthesis on failed or slow requests using a suite of advanced SRE theories:
  - **H1: Token Size Drives Latency** (Correlations between token counts and time)
  - **H2: KV Cache Fragmentation** (Impact of over-provisioning maxOutputTokens)
  - **H3: Prefill vs. Decode Context Bloat** (Isolating non-linear input vs. verbose output latency)
  - **H4: Agent Orchestration Overhead** (Latency dominated by handoffs vs. raw LLM generation)
  - **H5: Excessive Tool Iteration** (Detecting autonomous reasoning loops)
  - **H6: Data Scaling Degradation in Tools** (Tracking external tool degradation over time)
  - **H7: Request Queuing** (Detecting rate limits from concurrent request bursts)
  - **H8: Tool Failure Cost / Reliability** (Identifying tools with <95% success or extreme P95 execution times)
  - **H9: High Error Rate Impact** (Correlating QUOTA/TIMEOUT errors to latency spikes)
  - **H10: Configuration Drift** (Unannounced changes shifting the baseline)
  - **H11: The Reasoning Budget Overhead** (Isolating Gemini 2.5+ 'thinking' loop latency vs. output)
  - **H12: Systemic Daily Degradation** (Peak-hour variances or multi-day degradation)
  - **H13: Sub-optimal LLM Parameters** (Issues caused by specific temperature/top_k settings)
- **The Quality Flywheel**: Automatically curating "Golden Datasets" of successful agent interactions and exporting them to Vertex AI Eval for continuous offline testing and fine-tuning.
- **Sentiment & Quality Analysis**: Evaluating user satisfaction and agent response quality using BigQuery ML / LLM scoring (e.g., automated CSAT).
- **Agentic Cost & ROI Tracking**: Monitoring the financial cost per conversation and tracking prompt vs. completion ratios to catch runaway reasoning loops.
- **Cost Attribution & Chargeback**: Tracking granular costs per session, user, or agent for accurate billing and resource allocation.
- **Responsible AI (RAI) & Safety Monitoring**: Tracking policy violations and hallucination/grounding rates within agent responses.
- **Security & Governance (Threat Detection)**: Proactively scanning user inputs for prompt injection, jailbreaking, or malicious intents using AI endpoints in BigQuery.
- **Semantic Search & Memory**: Converting conversation logs into vector embeddings for similarity-based knowledge retrieval and agent context.
- **Structured Memory Extraction**: Abstracting key facts and user intents from unstructured conversational logs into JSON format.
- **Predictive Analytics**: Training ML models to predict latency from request features using BigQuery ML.
- **Causal Inference**: Moving beyond simple correlation (e.g., Token Size determines Latency) to strict causation analysis via integrated A/B tests proving direct impact.
- **Multi-Project Analysis**: Scaling analysis to comparatively benchmark performance across entirely different environments or GCP projects.
- **Real-Time Alerting**: Integrating with monitoring platforms for proactive, zero-delay incident response.
- **Automated Remediation**: Closing the full loop from diagnosis to recommendation to automatically deploying real-time hotfixes.

## Generate Observability Reports

You can generate comprehensive performance and latency intelligence reports using the Observability Analyst Agent using a number of Playbooks

### Playbook: overview (Default System Overview)
By default, the agent executes the `overview` playbook, which provides a straightforward aggregation of latency and error metrics for the specified `--time_period` without attempting to mathematically compare it to a historical baseline.
```shell
tools/run_observability_analyst.sh --time_period 24h
```

### Playbook: health (Healthcheck)
By default, the agent evaluates the **last 24 hours (Current Reality)** against a sturdy **7-day Historical Baseline**.
*(Note: You can explicitly force the agent into any Playbook using `--playbook overview`, `--playbook health`, `--playbook incident`, `--playbook trend`, or `--playbook latest`)*
```shell
tools/run_observability_analyst.sh --playbook health
```

### Playbook: incident (Incident Review)
The `incident` playbook is designed for investigating isolated events, sudden latency spikes, or validating recent hotfixes within a tight, custom time window.

```shell
tools/run_observability_analyst.sh --playbook incident --time_period 6h --baseline_period 6h
```
**How it works**:
1. **Time-Shifted Baselines**: By specifying identical `time_period` and `baseline_period` (e.g. 6h), the LLM will automatically perform a "Time Shift." It will explicitly calculate non-overlapping bounds to evaluate the current 6-hour incident window against the 6 hours *immediately preceding* the incident.
2. **Deep Concurrency Investigation**: During an incident, the agent is proactively instructed to utilize `analyze_trace_concurrency` and `detect_sequential_bottlenecks` on the slowest anomalous traces. This mathematically proves if your multi-agent architecture is suffering from blocking execution or deadlocks during load.
3. **Hotfix Verification**: It utilizes the `get_latest_queries` tool to fetch the absolute most recent requests *inside* your defined incident window to verify if recent iterations or deployments are actively improving the situation.

### Playbook: latest (Single Trace Deep Dive)
The `latest` playbook acts as a microscope, giving you an end-to-end "X-Ray" of the single most recent root agent execution trace. It extracts exact tool sequential timing, concurrency proof, backwards compatibility with baselines, and root cause analysis.
```shell
tools/run_observability_analyst.sh --playbook latest
```

### Playbook: trend (Trend Analysis)
To evaluate long term structural degradation or improvement, you can pass a large timeframe and split it into chronological buckets. The agent will calculate slopes and trends over time:
```shell
# Analyzes the last 30 days of data, grouped into daily buckets
tools/run_observability_analyst.sh --playbook trend --time_period 30d --bucket_size 1d
```
**How it works**: The `trend` playbook explicitly uses the `--time_period` (e.g. `30d`) to establish the overall boundary of the time-series array, and `--bucket_size` (e.g. `1d`) to chop that boundary into discrete chronological steps. The LLM iterates over those 30 distinct daily buckets to mathematically calculate if the p95 slope is rising (degrading) or falling (improving) over the course of the month.
*(Note: Unlike the `incident` playbook, `trend` does **not** perform any time-shifting. It evaluates the exact historical block of data as specified by the `--time_period`).*
